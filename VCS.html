🗂️ Version Control কী এবং কেন ব্যবহার করবে?

প্রতিটি ডেভেলপারের সবচেয়ে গুরুত্বপূর্ণ টুল হলো Version Control System। এটি ছাড়া পেশাদার সফটওয়্যার ডেভেলপমেন্ট কল্পনাই করা যায় না।

🔷 What is Version Control? — Version Control কী?

Version Control System (VCS) হলো এমন একটি সিস্টেম যা সময়ের সাথে ফাইলের প্রতিটি পরিবর্তন রেকর্ড করে রাখে। ফলে তুমি যেকোনো সময় পুরনো যেকোনো সংস্করণে ফিরে যেতে পারো, কে কী পরিবর্তন করেছে দেখতে পারো, এবং একসাথে একাধিক মানুষ একই প্রজেক্টে কাজ করতে পারে কোনো ঝামেলা ছাড়াই।

সহজ ভাষায় বলতে গেলে — VCS হলো তোমার কোডের "Time Machine"। যেকোনো সময় অতীতে ফিরে যেতে পারবে, ভুলগুলো ঠিক করতে পারবে।

VCS ছাড়া ডেভেলপারদের কী হয়?

VCS শেখার আগে অধিকাংশ নতুন ডেভেলপার এই ভুলটি করেন:

my-project/ ├── index.html ├── index_backup.html ├── index_old.html ├── index_v2.html ├── index_final.html ├── index_FINAL_v2.html ├── index_FINAL_LAST.html ← 😅 এটাই কি আসল ফাইল? └── index_FINAL_LAST_copy.html ← নাকি এটা? style.css style_backup.css style_working.css ← কোনটা ব্যবহার করছি?
⚠️ VCS ছাড়া যে ৫টি বড় সমস্যা হয়:

১. ইতিহাস হারানো: কোড নষ্ট হলে বা ভুল হলে আগের অবস্থায় ফিরে যাওয়ার কোনো উপায় নেই।
২. দলগত বিশৃঙ্খলা: দুজন একই ফাইলে কাজ করলে একজনের কাজ অন্যজনের কাজ মুছে দেয়।
৩. Accountability নেই: কে কখন কী পরিবর্তন করেছে জানা যায় না।
৪. Backup নেই: কম্পিউটার নষ্ট হলে বা চুরি হলে সব কাজ শেষ।
৫. Experiment করতে ভয়: নতুন কিছু চেষ্টা করলে পুরনো কোড নষ্ট হওয়ার ভয় থাকে।

🔷 Why use Version Control? — কেন ব্যবহার করবে?

VCS শেখা একটি বিনিয়োগ — একবার শিখলে সারাজীবন উপকার পাবে। পেশাদার ডেভেলপমেন্টে VCS জানা অবশ্যক শর্ত।

সুবিধা বিবরণ বাস্তব উদাহরণ
ইতিহাস সংরক্ষণ প্রতিটি পরিবর্তন রেকর্ড হয়, যেকোনো সময় ফিরে যাওয়া যায় গতকালের কোডে ফিরে যেতে একটি কমান্ড
দলগত কাজ একসাথে ১০০ জনও একই প্রজেক্টে কাজ করতে পারে Alice header, Bob footer — কোনো conflict নেই
Online Backup GitHub-এ push করলে কোড চিরতরে নিরাপদ ল্যাপটপ চুরি হলেও কোড GitHub-এ থাকে
নিরাপদ Experiment Branch-এ পরীক্ষা করো, main কোড নিরাপদ নতুন design try করো, পছন্দ না হলে মুছে দাও
Code Review অন্যরা তোমার কোড পরীক্ষা করে comment দিতে পারে Pull Request → সিনিয়র দেখে approve করে
Bug Tracking কোন commit-এ bug এলো খুঁজে বের করা যায় git bisect দিয়ে bug-এর সময় খুঁজে পাও
Release Management নির্দিষ্ট version tag করে রাখা যায় v1.0, v2.0 — যেকোনো version deploy করো
✅ পেশাদার পরিসংখ্যান: বিশ্বের শীর্ষ সব tech company — Google, Meta, Microsoft, Amazon — প্রত্যেকেই Git ব্যবহার করে। চাকরির interview-এ Git না জানলে প্রায়ই reject করা হয়। এটি শেখা optional নয়, অবশ্যক।

🔷 Git vs Other VCS — তুলনামূলক আলোচনা

Version Control System দুই ধরনের — Centralized এবং Distributed। আজকের দুনিয়ায় Git (Distributed) সবচেয়ে জনপ্রিয়।

Centralized VCS (CVCS) পুরনো পদ্ধতি

একটি কেন্দ্রীয় সার্ভারে সব কোড থাকে। প্রত্যেক ডেভেলপার সেই সার্ভার থেকে কোড নামায় এবং সার্ভারে পাঠায়। SVN (Subversion) এবং CVS এই ধরনের VCS।

কেন্দ্রীয় সার্ভার (একমাত্র repository) ↑↓ ↑↓ ↑↓ Alice Bob Charlie (কাজ করে) (কাজ করে) (কাজ করে) সমস্যা: সার্ভার বন্ধ = কেউ কাজ করতে পারবে না! সমস্যা: ইন্টারনেট না থাকলে কিছুই করা যাবে না!

Distributed VCS (DVCS) আধুনিক পদ্ধতি

প্রতিটি ডেভেলপারের কম্পিউটারে সম্পূর্ণ repository থাকে। ইন্টারনেট ছাড়াও কাজ করা যায়। Git এবং Mercurial এই ধরনের।

GitHub (Remote/Cloud Backup) ↑↓ ↑↓ ↑↓ Alice Bob Charlie (সম্পূর্ণ (সম্পূর্ণ (সম্পূর্ণ repo আছে) repo আছে) repo আছে) সুবিধা: Offline-এ কাজ করা যায়! সুবিধা: যেকোনো কম্পিউটার থেকে full history দেখা যায়!
বিষয় Centralized (SVN) Distributed (Git)
Offline কাজ ❌ সম্ভব নয় ✅ সম্পূর্ণ সম্ভব
গতি ধীর (network-নির্ভর) দ্রুত (local)
History শুধু central server-এ সবার কম্পিউটারে
Branching ধীর ও জটিল দ্রুত ও সহজ
Backup একটি server নষ্ট = সব শেষ প্রত্যেকের কাছে copy আছে
জনপ্রিয়তা কমে যাচ্ছে বিশ্বে ৯৮%+ ডেভেলপার
✅ সরাসরি কথা: ২০২৬ সালে Git শেখা মানে শিল্পের মানদণ্ড শেখা। SVN বা অন্য VCS শেখার কোনো দরকার নেই — সরাসরি Git দিয়ে শুরু করো।

🐙 Git পরিচিতি ও ইনস্টলেশন

Git কী, কীভাবে কাজ করে এবং তোমার কম্পিউটারে কীভাবে ইনস্টল করবে — সবকিছু এখানে।

🔷 What is Git? — Git কী?

Git হলো বিশ্বের সবচেয়ে জনপ্রিয় Distributed Version Control System। ২০০৫ সালে Linux-এর জনক Linus Torvalds মাত্র কয়েক সপ্তাহে এটি তৈরি করেছিলেন। আজ বিশ্বের ৯৮%+ ডেভেলপার Git ব্যবহার করেন।

Git তোমার প্রজেক্টের প্রতিটি পরিবর্তনের সম্পূর্ণ ইতিহাস রাখে। কে, কখন, কী পরিবর্তন করেছে এবং কেন করেছে — সব কিছু সংরক্ষিত থাকে।

✅ Git ≠ GitHub:
Git = একটি সফটওয়্যার যা তোমার কম্পিউটারে চলে (Free, Open Source)
GitHub = একটি ওয়েবসাইট যেখানে Git repository online-এ রাখা যায় (Microsoft-এর owned)

Git ছাড়া GitHub চলে না, কিন্তু Git GitHub ছাড়াও একা কাজ করতে পারে।

Git-এর ইতিহাস — একটু গল্প

সাল ঘটনা
২০০৫ Linus Torvalds Linux kernel-এর জন্য Git তৈরি করেন। কারণ ছিল BitKeeper (তখনকার VCS) বিনামূল্যে ব্যবহারের অনুমতি বাতিল হয়ে যাওয়া।
২০০৮ GitHub চালু হয় — Git repository-র জন্য web-based platform।
২০১১ GitLab চালু হয় — self-hosted alternative।
২০১৮ Microsoft GitHub কিনে নেয় $7.5 billion ডলারে।
২০২৩+ GitHub-এ ১০০ মিলিয়নেরও বেশি ডেভেলপার। বিশ্বের সবচেয়ে বড় code hosting platform।

🔷 Installing Git Locally — Git ইনস্টল করা

Git তোমার অপারেটিং সিস্টেম অনুযায়ী আলাদাভাবে ইনস্টল করতে হয়। একবার ইনস্টল করলেই হয়, বারবার করতে হয় না।

Windows-এ ইনস্টল Windows

git-scm.com থেকে Windows installer ডাউনলোড করো। ইনস্টলের সময় সব ডিফল্ট option রেখে Next করতে থাকো। ইনস্টলের পর Git Bash নামে একটি terminal পাবে — এটিই ব্যবহার করবে।

ডাউনলোড: https://git-scm.com/download/win ইনস্টলের পর দুটো জিনিস পাবে: → Git Bash (Unix-style terminal — এটি ব্যবহার করো) → Git GUI (Graphical interface — optional) VS Code-এ Terminal খুলতে: Ctrl + ` (এটিও Git Bash হিসেবে কাজ করবে)

macOS-এ ইনস্টল macOS

# পদ্ধতি ১: Homebrew দিয়ে (সবচেয়ে সহজ) brew install git # পদ্ধতি ২: Xcode Command Line Tools xcode-select --install # (এতে Git সহ আসে) # পদ্ধতি ৩: Official installer # https://git-scm.com/download/mac

Linux-এ ইনস্টল Linux

# Ubuntu / Debian sudo apt update sudo apt install git # Fedora / RHEL sudo dnf install git # Arch Linux sudo pacman -S git

ইনস্টল যাচাই করো

# Terminal বা Git Bash খুলে লেখো: git --version # সফল হলে দেখাবে: git version 2.43.0 ✅ # এই আউটপুট দেখলে Git সফলভাবে ইনস্টল হয়েছে!
✅ VS Code ব্যবহারকারীদের জন্য: VS Code-এ Git built-in আছে। Terminal-এ git কমান্ড চলবে। Source Control panel (Ctrl+Shift+G) দিয়ে graphical-ভাবেও Git ব্যবহার করা যায়। তবে command line শেখা জরুরি — এতে সব কিছু বোঝা যায়।

⚙️ git config — প্রথমবারের Setup

Git ইনস্টলের পর একবার কনফিগার করতে হয়। এই তথ্য প্রতিটি commit-এ যোগ হয়।

🔷 Local vs Global Config — কনফিগারেশনের ধরন

Git কনফিগারেশন তিন স্তরে কাজ করে। উপরের স্তর নিচের স্তরকে override করে।

স্তর কোথায় সংরক্ষিত কখন প্রযোজ্য flag
System /etc/gitconfig কম্পিউটারের সব user-এ --system
Global ~/.gitconfig তোমার সব প্রজেক্টে --global
Local .git/config শুধু এই একটি প্রজেক্টে (default)
✅ সাধারণত --global ব্যবহার করো — এতে একবার সেট করলেই তোমার কম্পিউটারের সব প্রজেক্টে কাজ করবে। ভিন্ন প্রজেক্টে ভিন্ন নাম/ইমেইল দরকার হলে সেই ফোল্ডারে গিয়ে --local দিয়ে সেট করো।

🔷 Repository Initialization ও প্রথমবারের Config

নাম ও ইমেইল সেট করো অবশ্যই করতে হবে

প্রতিটি commit-এ author-এর নাম ও ইমেইল সংরক্ষিত হয়। GitHub-এর যে ইমেইল ব্যবহার করবে সেটাই এখানে দাও।

# নাম সেট করো git config --global user.name "তোমার পুরো নাম" # ইমেইল সেট করো (GitHub-এর ইমেইল দাও) git config --global user.email "email@example.com"

অন্যান্য গুরুত্বপূর্ণ Config দরকারি

# Default branch-এর নাম 'main' করো (modern standard) git config --global init.defaultBranch main # রঙিন output-এর জন্য (terminal সুন্দর দেখাবে) git config --global color.ui auto # Default editor VS Code করো git config --global core.editor "code --wait" # Windows-এ line ending সমস্যা এড়াতে git config --global core.autocrlf true # macOS/Linux-এ git config --global core.autocrlf input # Push এর সময় current branch-ই push হোক git config --global push.default current

Aliases — সময় বাঁচানোর শর্টকাট

বারবার একই লম্বা কমান্ড টাইপ করতে হয়। Alias হলো সেই কমান্ডের ছোট ডাকনাম। একবার তৈরি করো, সারাজীবন সময় বাঁচাও।

# সংক্ষিপ্ত কমান্ড তৈরি করো git config --global alias.st status git config --global alias.co checkout git config --global alias.br branch git config --global alias.lg "log --oneline --graph --decorate --all" git config --global alias.last "log -1 HEAD --stat" git config --global alias.unstage "restore --staged" # ব্যবহার: git st # → git status git co main # → git checkout main git br # → git branch git lg # → সুন্দর tree-style log git last # → শেষ commit-এর বিবরণ git unstage . # → সব unstage করো

সব Config দেখো ও যাচাই করো

# সব config দেখো git config --list # নির্দিষ্ট config দেখো git config user.name git config user.email # config ফাইল সরাসরি দেখো/সম্পাদনা করো cat ~/.gitconfig # আউটপুট উদাহরণ: [user] name = ফরীদুর রহমান email = farid@example.com [init] defaultBranch = main [color] ui = auto [alias] st = status co = checkout

📁 What is a Repository — Repository তৈরি

Repository হলো তোমার প্রজেক্টের সম্পূর্ণ ইতিহাস সহ সব ফাইলের ভান্ডার।

🔷 What is a Repository? — Repository কী?

Repository (সংক্ষেপে repo) হলো একটি বিশেষ ফোল্ডার যেখানে তোমার প্রজেক্টের সব ফাইল এবং সেই ফাইলগুলোর সম্পূর্ণ পরিবর্তনের ইতিহাস সংরক্ষিত থাকে। প্রতিটি repository-র ভেতরে একটি লুকানো .git ফোল্ডার থাকে। এই .git ফোল্ডারটিই হলো আসল repository — এখানেই সব তথ্য জমা থাকে।

my-website/ ← এটি তোমার প্রজেক্ট ফোল্ডার ├── .git/ ← 🔒 Git-এর তথ্যভান্ডার (কখনো মুছবে না!) │ ├── HEAD ← বর্তমান branch কোনটি │ ├── config ← এই repo-র local config │ ├── objects/ ← সব commit, file data এখানে │ └── refs/ │ ├── heads/ ← সব local branch │ └── tags/ ← সব tag ├── index.html ├── style.css └── script.js
⚠️ .git ফোল্ডার কখনো মুছবে না! এই ফোল্ডারটি মুছে ফেললে সব commit history, branch, tag — সব চলে যাবে। প্রজেক্ট ফোল্ডার থাকবে কিন্তু Git আর কাজ করবে না।

🔷 git init — নতুন Repository শুরু করা

git init কমান্ড দিলে বর্তমান ফোল্ডারে একটি নতুন Git repository তৈরি হয়। এটি .git ফোল্ডার তৈরি করে এবং Git tracking শুরু করে।

নতুন প্রজেক্টে Git শুরু করা সবচেয়ে সাধারণ

# ধাপ ১: নতুন ফোল্ডার তৈরি করো mkdir my-website cd my-website # ধাপ ২: Git repository শুরু করো git init # আউটপুট: # Initialized empty Git repository in /path/to/my-website/.git/ # ধাপ ৩: কিছু ফাইল তৈরি করো echo "<!DOCTYPE html><html><body>হ্যালো!</body></html>" > index.html # ধাপ ৪: প্রথম commit করো git add . git commit -m "feat: প্রজেক্ট শুরু করা হয়েছে"

বিদ্যমান প্রজেক্টে Git যোগ করা পুরনো প্রজেক্ট

# পুরনো প্রজেক্ট ফোল্ডারে যাও cd existing-project # Git tracking শুরু করো git init # সব ফাইল প্রথম commit-এ যোগ করো git add . git commit -m "feat: পুরনো প্রজেক্টে Git যোগ করা হয়েছে"

🔷 git clone — GitHub থেকে Repository নামানো

git clone দিয়ে GitHub বা যেকোনো remote repository-র সম্পূর্ণ কপি — সব commit history, branch, tag সহ — তোমার কম্পিউটারে নামিয়ে আনা যায়।

HTTPS দিয়ে Clone শুরুতে এটি করো

# সাধারণ clone git clone https://github.com/username/repo-name.git # নির্দিষ্ট ফোল্ডার নামে clone git clone https://github.com/username/repo-name.git my-project # Clone শেষে ফোল্ডারে ঢোকো cd repo-name

SSH দিয়ে Clone SSH Key থাকলে

# SSH দিয়ে clone (password লাগে না, বেশি নিরাপদ) git clone git@github.com:username/repo-name.git # নির্দিষ্ট branch clone করো git clone --branch develop https://github.com/username/repo.git # Shallow clone — শুধু সর্বশেষ snapshot (দ্রুত) git clone --depth=1 https://github.com/username/repo.git
✅ Clone করার পর কী পাবে:
  • সম্পূর্ণ প্রজেক্ট ফাইল
  • সম্পূর্ণ commit history
  • সব branch-এর তথ্য
  • origin নামে remote automatically সেট হয়ে যায়

🗃️ Working Directory, Staging Area — Git-এর তিনটি অঞ্চল

Git সঠিকভাবে বোঝার সবচেয়ে গুরুত্বপূর্ণ ধারণা — ফাইল তিনটি অঞ্চলের মধ্যে যাতায়াত করে।

🔷 Git-এর তিনটি অঞ্চল

Git-এ যেকোনো ফাইল সবসময় তিনটির মধ্যে একটি অঞ্চলে থাকে। এই তিনটি অঞ্চল বোঝাটা Git শেখার সবচেয়ে গুরুত্বপূর্ণ ধাপ। এটি না বুঝলে পরের সব বিষয় কঠিন লাগবে।

অঞ্চল বাংলায় কী হয় এখানে? কমান্ড
Working Directory কাজের জায়গা তুমি এখানে ফাইল তৈরি, সম্পাদনা, মুছে ফেলো। Git ট্র্যাক করে কিন্তু save করে না। সাধারণ ফাইল কাজ
Staging Area প্রস্তুতির মঞ্চ পরবর্তী commit-এ কোন কোন পরিবর্তন নেবে তা বেছে নাও। এটিকে "Index" বলেও জানে। git add
Repository (.git) ইতিহাস ভান্ডার Commit হলে পরিবর্তন এখানে চিরতরে সংরক্ষিত হয়। এখান থেকে সরানো কঠিন। git commit

ফটোগ্রাফারের উপমায় বোঝো

✅ উপমা — Photo Album বানানো:

Working Directory = ফটো তোলার সেশন (তুমি ছবি তুলছো)
Staging Area = কোন ছবিগুলো album-এ রাখবে বেছে নাও
Repository = Photo Album (বেছে নেওয়া ছবি চিরতরে সংরক্ষিত)

একটি ফটো সেশনে ১০০টি ছবি তুললেও সবগুলো album-এ রাখা হয় না — ভালো ছবিগুলো বেছে নেওয়া হয়। Staging Area-ও ঠিক তাই করে।

ফাইলের যাত্রা — step by step

# ধাপ ১: Working Directory-তে ফাইল তৈরি করো echo "হ্যালো" > index.html # ফাইলটি এখন "Untracked" অবস্থায় → Working Directory-তে # ধাপ ২: Staging Area-তে যোগ করো (বেছে নাও) git add index.html # ফাইলটি এখন "Staged" → Staging Area-তে # ধাপ ৩: Repository-তে সংরক্ষণ করো git commit -m "feat: প্রথম ফাইল তৈরি" # ফাইলটি এখন "Committed" → Repository-তে ✅ # বর্তমান অবস্থা সবসময় দেখতে পারো: git status

🔷 ফাইলের পাঁচটি অবস্থা — File States

Git-এ প্রতিটি ফাইল পাঁচটির মধ্যে একটি অবস্থায় থাকে। git status দিলে বর্তমান অবস্থা দেখা যায়।

অবস্থা অর্থ git status-এ দেখায় পরের পদক্ষেপ
Untracked নতুন ফাইল — Git এখনো জানে না এটার কথা Untracked files: index.html (লাল) git add index.html
Modified Git চেনে, কিন্তু নতুন পরিবর্তন stage হয়নি Changes not staged: style.css (লাল) git add style.css
Staged Staging Area-তে, commit-এর জন্য প্রস্তুত Changes to be committed: index.html (সবুজ) git commit -m "..."
Committed Repository-তে চিরতরে সংরক্ষিত, নিরাপদ nothing to commit কাজ শেষ ✅
Ignored .gitignore-এ আছে, Git সম্পূর্ণ উপেক্ষা করে দেখায় না কোনো পদক্ষেপ নেই

🔷 git add — Staging Area-তে যোগ করা

git add দিয়ে তুমি Git-কে বলো "এই পরিবর্তনগুলো পরবর্তী commit-এ রাখতে চাই।" এটি Staging Area-তে ফাইল যোগ করে।

কমান্ড কাজ কখন ব্যবহার করবে?
git add index.html একটি নির্দিষ্ট ফাইল stage করে শুধু একটি ফাইলের পরিবর্তন নিতে
git add style.css app.js একসাথে একাধিক ফাইল কয়েকটি নির্দিষ্ট ফাইল
git add . বর্তমান ফোল্ডারের সব পরিবর্তন সব পরিবর্তন একসাথে নিতে
git add -A সব পরিবর্তন (deleted সহ) .-এর মতো কিন্তু deleted ফাইলও নেয়
git add *.css সব .css ফাইল নির্দিষ্ট extension-এর সব ফাইল
git add css/ css/ ফোল্ডারের সব পরিবর্তন একটি ফোল্ডারের সব
git add -p Interactive — নির্দিষ্ট অংশ stage করো একই ফাইলের কিছু অংশ নিতে

বাস্তব উদাহরণ — আলাদা commit-এর জন্য আলাদা add

# তুমি ৩টি আলাদা কাজ করেছো: # ১. index.html-এ নতুন section যোগ # ২. style.css-এ navbar রঙ পরিবর্তন # ৩. script.js-এ form validation # এগুলো ৩টি আলাদা commit করা উচিত: # প্রথম commit: HTML পরিবর্তন git add index.html git commit -m "feat: হোম পেজে service section যোগ করা হয়েছে" # দ্বিতীয় commit: CSS পরিবর্তন git add style.css git commit -m "style: navbar-এর রঙ পরিবর্তন করা হয়েছে" # তৃতীয় commit: JS পরিবর্তন git add script.js git commit -m "feat: যোগাযোগ ফর্মে validation যোগ করা হয়েছে"
✅ একটি commit = একটি কারণ। "নেভিগেশন ঠিক, footer যোগ, login page বানানো" — এটি ৩টি আলাদা commit হওয়া উচিত। ছোট ছোট commit করলে ভবিষ্যতে bug খোঁজা এবং নির্দিষ্ট পরিবর্তন revert করা সহজ হয়।

📸 Committing Changes — পরিবর্তন সংরক্ষণ

Commit হলো Git-এর মূল একক — প্রতিটি commit একটি সম্পূর্ণ snapshot।

🔷 Commit কী? — সহজ ব্যাখ্যা

Commit হলো তোমার প্রজেক্টের একটি নির্দিষ্ট মুহূর্তের সম্পূর্ণ snapshot — সেই মুহূর্তে সব ফাইল কেমন ছিল তার সম্পূর্ণ ছবি। প্রতিটি commit একটি unique ID (SHA hash) পায়।

প্রতিটি commit-এ থাকে:

অংশ উদাহরণ কাজ
SHA Hash a1b2c3d4e5f6... প্রতিটি commit-এর unique ID (40 character)
Author ফরীদুর রহমান <farid@example.com> কে commit করেছে
Date Mon Jun 2 10:30:00 2026 +0600 কখন commit করেছে
Message feat: নেভিগেশন বার যোগ করা হয়েছে কী পরিবর্তন করেছে
Parent Hash আগের commit-এর SHA ইতিহাসের শৃঙ্খল তৈরি করে
Snapshot সব ফাইলের বিষয়বস্তু সেই মুহূর্তের সম্পূর্ণ প্রজেক্ট
⚠️ Commit = Save বোতাম নয়!
Ctrl+S দিয়ে ফাইল save করা এবং git commit করা সম্পূর্ণ আলাদা জিনিস। Ctrl+S করলে ফাইল disk-এ save হয়, কিন্তু Git track করে না। Git-এ সংরক্ষণ করতে হলে git add + git commit করতে হবে।

🔷 git commit — বিভিন্ন পদ্ধতি

# সাধারণ commit (সবচেয়ে বেশি ব্যবহৃত) git commit -m "feat: নেভিগেশন বার যোগ করা হয়েছে" # Stage + commit একসাথে # (শুধু already tracked ফাইলের জন্য — নতুন ফাইলে কাজ করবে না) git commit -am "style: হেডারের রঙ পরিবর্তন করা হয়েছে" # বিস্তারিত message-এর জন্য editor খুলবে git commit # (VS Code বা Vim খুলবে — লিখে save করলে commit হবে) # শেষ commit-এ ছোট পরিবর্তন যোগ করো (message একই রাখো) git add missed-file.html git commit --amend --no-edit # শেষ commit-এর message পরিবর্তন করো git commit --amend -m "feat: নতুন সঠিক message"
⚠️ --amend সাবধানে ব্যবহার করো! git commit --amend শেষ commit পরিবর্তন করে নতুন commit তৈরি করে। যদি সেই commit ইতিমধ্যে GitHub-এ push করা হয়ে থাকে, তাহলে amend করা এড়াও — অন্যদের সমস্যা হতে পারে।

🔷 Commit Message লেখার নিয়ম — Best Practice

Commit message হলো ভবিষ্যতের তোমাকে এবং তোমার সহকর্মীদের জন্য চিঠি। ভালো message লেখা একটি professional অভ্যাস যা বড় প্রজেক্টে অপরিহার্য।

Conventional Commits — শিল্পের মানদণ্ড

Conventional Commits হলো commit message-এর একটি জনপ্রিয় মান। Format: type(scope): description

type কখন ব্যবহার? উদাহরণ
feat: নতুন ফিচার যোগ feat: লগইন পেজ তৈরি করা হয়েছে
fix: Bug ঠিক করা fix: মোবাইলে মেনু বন্ধ না হওয়ার bug ঠিক
style: শুধু CSS/formatting পরিবর্তন style: বাটনের রঙ ও padding পরিবর্তন
docs: Documentation পরিবর্তন docs: README-তে ইনস্টলেশন গাইড
refactor: কোড পুনর্গঠন (ফিচার একই) refactor: getUserData ফাংশন পুনর্লিখন

🙈 .gitignore — ফাইল উপেক্ষা করা

কিছু ফাইল Git-এ রাখা উচিত নয় — password, auto-generated ফাইল, editor ফাইল। .gitignore দিয়ে সেগুলো বাদ দাও।

🔷 .gitignore কী এবং কেন লাগে?

.gitignore হলো একটি সাধারণ text ফাইল যেখানে তুমি বলে দাও কোন ফাইল বা ফোল্ডার Git track করবে না।

ধরন উদাহরণ কেন ignore করবে?
গোপন তথ্য .env, config.secret.js Password, API key — GitHub-এ গেলে বিপদ!
Auto-generated node_modules/, dist/, build/ পুনরায় তৈরি করা যায়, commit করার দরকার নেই
OS ফাইল .DS_Store (Mac), Thumbs.db (Win) ব্যক্তিগত system ফাইল, অন্যদের দরকার নেই
Editor ফাইল .vscode/, .idea/ ব্যক্তিগত editor settings
Log ফাইল *.log, npm-debug.log Runtime-এ তৈরি হয়, repository-তে দরকার নেই

📜 Commit History — git log বিস্তারিত

Commit history পড়তে পারা মানে প্রজেক্টের পুরো গল্প বুঝতে পারা।

🔷 git log — সব commit দেখো

# বিস্তারিত log (সব তথ্য) git log # সংক্ষিপ্ত — এক লাইনে প্রতিটি commit git log --oneline # Tree আকারে সব branch (সবচেয়ে বেশি ব্যবহৃত) git log --oneline --graph --decorate --all # শেষ n টি commit git log -5 git log -10 # নির্দিষ্ট ফাইলের history git log -- style.css git log --oneline -- index.html # নির্দিষ্ট ব্যক্তির commit git log --author="Farid" # নির্দিষ্ট সময়ের মধ্যে git log --since="2026-01-01" --until="2026-06-01" git log --since="3 days ago" git log --since="1 week ago" # Commit message-এ keyword খোঁজো git log --grep="navbar" git log --grep="fix" # কোড-এ নির্দিষ্ট text কোথায় যোগ/বাদ হয়েছে git log -S "getUserData" ← "pickaxe" search
$ git log --oneline --graph --all * a1b2c3d (HEAD -> main, origin/main) Merge: feature/contact তৈরি |\ | * b2c3d4e (feature/contact) feat: contact form validation যোগ | * c3d4e5f feat: contact page HTML তৈরি * | d4e5f6g fix: mobile navbar bug ঠিক * | e5f6g7h style: footer redesign |/ * f6g7h8i feat: about page তৈরি * g7h8i9j chore: প্রজেক্ট শুরু # মানে: # HEAD → তুমি এখন main-এ আছো # * = commit # |\ = branch diverge হয়েছে # |/ = branch merge হয়েছে

↩️ পরিবর্তন Undo করা — git restore, revert, reset

Git-এর সবচেয়ে শক্তিশালী দিক হলো যেকোনো পরিবর্তন undo করার ক্ষমতা। কিন্তু ভুল কমান্ড দিলে সমস্যা হতে পারে।

🔷 পরিস্থিতি অনুযায়ী সঠিক কমান্ড

পরিস্থিতি সঠিক কমান্ড নিরাপদ? পরিবর্তন হারায়?
Stage করা file unstage করো git restore --staged file.html ✅ না — পরিবর্তন Working Dir-এ থাকে
Working Dir-এর পরিবর্তন বাতিল git restore file.html ⚠️ হ্যাঁ — forever মুছে যায়
সব unstaged পরিবর্তন বাতিল git restore . ⚠️ হ্যাঁ — সব মুছে
Commit নিরাপদে বাতিল (new commit) git revert a1b2c3d ✅ সবচেয়ে নিরাপদ না — history রেকর্ড থাকে
শেষ commit বাতিল, staged রাখো git reset --soft HEAD~1 ✅ না — Staging Area-তে থাকে
শেষ commit বাতিল, unstaged রাখো git reset HEAD~1 ⚠️ না — Working Dir-এ থাকে
শেষ commit + সব পরিবর্তন মুছো git reset --hard HEAD~1 ❌ বিপজ্জনক হ্যাঁ — চিরতরে

🔷 git restore — Working Directory ও Staging Area ঠিক করা

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # পরিস্থিতি ১: index.html edit করেছো, পুরনো version চাই # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git restore index.html # ⚠️ Working Dir-এর পরিবর্তন চিরতরে মুছে যাবে! # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # পরিস্থিতি ২: git add করেছো, কিন্তু এই commit-এ নেবে না # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git restore --staged style.css # ✅ Unstaged হলো, পরিবর্তন Working Dir-এ থাকলো # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # পরিস্থিতি ৩: নির্দিষ্ট commit থেকে file ফিরিয়ে আনো # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git restore --source=a1b2c3d style.css # a1b2c3d commit-এর style.css ফিরে আসবে Working Dir-এ

🔷 git revert — নিরাপদভাবে Commit বাতিল

git revert হলো সবচেয়ে নিরাপদ "undo" পদ্ধতি। এটি পুরনো commit মুছে না, বরং সেটার বিপরীত একটি নতুন commit তৈরি করে। Public/shared branch-এ সবসময় git revert ব্যবহার করো।

# শেষ commit বাতিল করো (নতুন "Revert" commit তৈরি হবে) git revert HEAD # নির্দিষ্ট commit বাতিল করো git revert a1b2c3d # একাধিক commit বাতিল করো git revert HEAD~3..HEAD # Interactive নয়, সরাসরি commit করো git revert --no-edit a1b2c3d # History-তে কী হয়: # আগে: A → B → C (C-তে bug আছে) # revert: A → B → C → D (D = C-এর বিপরীত, bug চলে গেছে) # C-এর history রেকর্ড থাকে!

🔷 git reset — History পরিবর্তন (সাবধানে)

git reset commit history পরিবর্তন করে। তিনটি mode আছে:

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # --soft: commit বাতিল, পরিবর্তন Staged থাকে # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git reset --soft HEAD~1 # ব্যবহার: commit message ভুল ছিল, আবার commit করতে চাই # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # --mixed (default): commit বাতিল, পরিবর্তন Unstaged থাকে # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git reset HEAD~1 # ব্যবহার: commit ভুল ছিল, পরিবর্তন আবার review করতে চাই # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # --hard: commit + সব পরিবর্তন মুছে দাও # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git reset --hard HEAD~1 # ব্যবহার: পুরো experiment নষ্ট, সব বাতিল চাই # শেষ ৩টি commit বাতিল git reset --soft HEAD~3 ← পরিবর্তন রেখে git reset --hard HEAD~3 ← সব মুছে # নির্দিষ্ট commit-এ ফিরে যাও git reset --hard a1b2c3d
⚠️ গুরুত্বপূর্ণ সতর্কতা:
Push করার পরে git reset করো না! GitHub-এ push করা commit reset করলে অন্যদের কাজ নষ্ট হতে পারে এবং force push দরকার পড়বে। Push হয়ে গেলে সবসময় git revert ব্যবহার করো।

কিছু ফাইল কখনো Git-এ রাখা উচিত নয় — password, auto-generated files, editor settings। .gitignore এই কাজ করে।

🌿 Branching Basics — Branch তৈরি, Switch, Rename, Delete

Branch হলো Git-এর সবচেয়ে শক্তিশালী feature। এটা সঠিকভাবে শিখলে টিমওয়ার্ক অনেক সহজ হয়ে যায়।

🔷 Branch-এর মূল কমান্ড

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Branch দেখা # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # সব local branch git branch # * main ← * মানে এখন এই branch-এ আছো # feature/login # fix/navbar # সব branch (remote সহ) git branch -a # * main # feature/login # remotes/origin/main # remotes/origin/feature/login # Branch-এর শেষ commit দেখো git branch -v # * main a1b2c3d feat: navbar search # feature b2c3d4e feat: login page WIP
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Branch তৈরি করা # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # শুধু তৈরি (switch হবে না) git branch feature/login-page # তৈরি + switch একসাথে (সবচেয়ে বেশি ব্যবহৃত) git checkout -b feature/login-page # আধুনিক পদ্ধতি (Git 2.23+) git switch -c feature/login-page # নির্দিষ্ট commit থেকে branch তৈরি git checkout -b hotfix/payment a1b2c3d # Remote branch track করে local branch তৈরি git checkout -b feature/login origin/feature/login
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Branch Switch করা # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # পুরনো পদ্ধতি git checkout main git checkout feature/login # আধুনিক পদ্ধতি (পরিষ্কার, confusing নয়) git switch main git switch feature/login # আগের branch-এ ফিরে যাও (- মানে "আগেরটা") git switch - git checkout -
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Branch Rename করা # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # বর্তমান branch rename git branch -m new-name # নির্দিষ্ট branch rename git branch -m old-name new-name # Remote-এও rename করতে: git push origin --delete old-name ← পুরনো নাম মুছো git push origin -u new-name ← নতুন নামে push করো
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Branch Delete করা # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Merge হওয়া branch মুছো (নিরাপদ — merge না হলে error দেবে) git branch -d feature/login # Merge না হলেও মুছো (জোর করে) git branch -D feature/experiment # Remote branch মুছো git push origin --delete feature/login # Merge হওয়া সব branch একসাথে মুছো (main ও develop ছাড়া) git branch --merged | grep -v "\*\|main\|develop" | xargs git branch -d

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # Feature Branch Workflow — ধাপে ধাপে # ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # ধাপ ১: main-এ সর্বশেষ কোড আনো git checkout main git pull origin main # ধাপ ২: নতুন feature branch তৈরি করো git checkout -b feature/user-profile # ধাপ ৩: কাজ করো ও commit করো # ... profile.html তৈরি করলে ... git add profile.html git commit -m "feat: user profile page HTML structure" # ... CSS যোগ করলে ... git add style.css git commit -m "style: profile page সাজানো হয়েছে" # ধাপ ৪: GitHub-এ push করো (PR দেওয়ার জন্য) git push -u origin feature/user-profile # ধাপ ৫: GitHub-এ Pull Request দাও # → "Compare & pull request" ক্লিক করো # ধাপ ৬: Review ও merge-এর পর branch মুছো git checkout main git pull origin main ← merged কোড নামাও git branch -d feature/user-profile ← local মুছো

🔀 Merging Basics — Branch একত্রিত করা

Feature তৈরি শেষ হলে main-এ যোগ করার সময়। Merge বোঝা মানে Git-এর সবচেয়ে গুরুত্বপূর্ণ অংশ বোঝা।

🔷 Merge করার পদ্ধতি

# সাধারণ merge (সবচেয়ে সহজ) git checkout main git merge feature/login-page # Fast-forward merge হবে (কোনো diverge ছিল না) # Merge commit সহ (history পরিষ্কার রাখে, দলগত কাজে ভালো) git merge --no-ff feature/login-page -m "feat: login page main-এ যোগ করা হয়েছে" # Squash merge (feature-এর সব ছোট commit একটিতে মিলিয়ে) git merge --squash feature/login-page git commit -m "feat: login page — email/password validation সহ"

Fast-Forward vs No-Fast-Forward

Fast-Forward Merge (কোনো diverge নেই): আগে: main: A → B → C feature: A → B → C → D → E merge: main: A → B → C → D → E (শুধু pointer সরে গেল, কোনো merge commit নেই) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ No-Fast-Forward Merge (--no-ff): আগে: main: A → B → C → F feature: A → B → D → E merge: main: A → B → C → F → G (merge commit) ↗ E → D ↗ (feature branch-এর existence রেকর্ড থাকে)
Merge পদ্ধতি History কখন ব্যবহার
Fast-Forward Linear — simple ও clean ছোট, একা কাজ, personal project
No-Fast-Forward (--no-ff) Merge commit — feature-এর context রাখে দলগত কাজ, feature branch workflow
Squash সব commit একটিতে — cleanest PR merge, ছোট fix commit-এর দল

⚔️ Merge Conflict — কারণ, সমাধান ও প্রতিরোধ

Conflict দেখে ভয় পেও না। এটি সমাধানযোগ্য এবং সমাধান জানা একজন দক্ষ Developer-এর চিহ্ন।

🔷 Conflict কেন হয়?

যখন দুটো branch-এ একই ফাইলের একই স্থানে ভিন্ন পরিবর্তন হয়, Git নিজে সিদ্ধান্ত নিতে পারে না কোনটা রাখবে। তখন সে "Conflict" চিহ্নিত করে এবং তোমাকে সিদ্ধান্ত নিতে বলে।

পরিস্থিতি: তুমি main-এ: <h1>আমার ওয়েবসাইট</h1> পরিবর্তন করেছো সহকর্মী feature-এ: <h1>আমাদের ওয়েবসাইট</h1> পরিবর্তন করেছে এখন merge করলে Conflict!
Conflict হলে index.html-এ এই চিহ্ন দেখাবে: <<<<<<< HEAD <h1>আমার ওয়েবসাইট</h1> ← তোমার কোড (বর্তমান branch) ======= <h1>আমাদের ওয়েবসাইট</h1> ← আসছে কোড (merge করছো যেটা) >>>>>>> feature/update-title মানে: <<<<<<< HEAD → এটা বর্তমান branch (HEAD) এর কোড ======= → বিভাজক রেখা >>>>>>> → এটা incoming branch-এর কোড

🔷 Conflict সমাধানের ধাপ

# ধাপ ১: Conflict আছে কোন ফাইলে জানো $ git status # both modified: index.html # both modified: style.css # ধাপ ২: ফাইলটি খুলো এবং conflict markers খোঁজো # VS Code-এ রঙিন করে দেখাবে এবং বোতাম দেবে # ধাপ ৩: সিদ্ধান্ত নাও — কোনটা রাখবে বা দুটো মিলিয়ে নাও # Conflict-এর আগে: <<<<<<< HEAD <h1>আমার ওয়েবসাইট</h1> ======= <h1>আমাদের ওয়েবসাইট</h1> >>>>>>> feature/update-title # Conflict সমাধানের পরে (markers মুছে ঠিক কোড রাখো): <h1>আমাদের ওয়েবসাইট</h1> # ধাপ ৪: সমাধান করা ফাইল stage করো git add index.html git add style.css # ধাপ ৫: Merge সম্পন্ন করো git commit -m "fix: title conflict সমাধান — আমাদের রাখা হয়েছে"
# Merge বাতিল করতে চাইলে git merge --abort # সব পরিবর্তন বাতিল, merge-এর আগের অবস্থায় ফিরে যাবে # VS Code-এ Conflict সমাধান করা অনেক সহজ: # "Accept Current Change" → HEAD (তোমার) রাখো # "Accept Incoming Change" → আসছে পরিবর্তন রাখো # "Accept Both Changes" → দুটোই রাখো

🔷 Conflict প্রতিরোধের উপায়

🔷 git push — Remote-এ পাঠানো

# প্রথমবার push করো এবং tracking সেট করো git push -u origin main # এরপর থেকে শুধু: git push # নির্দিষ্ট branch push করো git push origin feature/login # সব tag push করো git push --tags # নির্দিষ্ট tag push করো git push origin v1.0.0 # Remote branch মুছো git push origin --delete feature/old-feature # Force push — ⚠️ সাবধানে git push --force-with-lease ← নিরাপদ force push git push --force ← বিপজ্জনক — avoid করো
⚠️ git push --force কখনো করো না shared branch-এ!
force push করলে অন্যদের commit history ধ্বংস হতে পারে। যদি করতেই হয়, --force-with-lease ব্যবহার করো — এটি অন্য কেউ push করলে সেটা আগে দেখে।

🔷 git pull ও git fetch — Remote থেকে আনা

কমান্ড কী করে কখন ব্যবহার
git fetch Remote-এর পরিবর্তন download করে, কিন্তু merge করে না দেখতে চাও কী এলো, সাথে সাথে merge নয়
git pull git fetch + git merge সহকর্মীর কাজ নামাও ও মিলিয়ে নাও
git pull --rebase git fetch + git rebase Clean linear history চাইলে
# সহকর্মীর কাজ নামাও git pull # নির্দিষ্ট branch pull করো git pull origin main git pull origin feature/login # নিরাপদ পদ্ধতি — দেখে তারপর merge করো git fetch origin ← আগে download করো git log origin/main --oneline -5 ← কী এলো দেখো git diff HEAD origin/main ← পার্থক্য দেখো git merge origin/main ← ভালো লাগলে merge করো # Deleted remote branch সরাও git fetch --prune git remote prune origin

🔁 Pull Requests — তৈরি, Review ও Merge

Pull Request (PR) হলো আধুনিক collaborative development-এর মূল পদ্ধতি। সরাসরি main-এ merge না করে PR দিলে quality নিশ্চিত হয়।

🔷 Pull Request কী এবং কেন দরকার?

Pull Request মানে — "আমার feature branch-এর পরিবর্তনগুলো main-এ নিয়ে নাও" এই formal অনুরোধ। এটি দিলে টিম সদস্যরা কোড review করতে পারে, test করতে পারে এবং approve-এর পরেই merge হয়।

🔷 PR Review Process

Review Action অর্থ ও কখন
Comment Approve/reject ছাড়া মন্তব্য। প্রশ্ন করা বা suggestion-এর জন্য।

🔷 Cherry Pick — নির্দিষ্ট Commit নেওয়া

📦 Git Stash — অসম্পূর্ণ কাজ সরিয়ে রাখা

কাজের মাঝে জরুরি কিছু করতে হলে Stash একটা lifesaver।

🔷 Stash কেন দরকার?

✅ বাস্তব পরিস্থিতি:
তুমি login form-এর কাজ অর্ধেক করেছো। হঠাৎ ফোন — "Production-এ payment crash করছে! এখনই ঠিক করো!"

সমস্যা: অর্ধেক কাজ commit করার মতো অবস্থায় নেই।
সমাধান: git stash করো → main-এ যাও → hotfix করো → ফিরে আসো → git stash pop!
# কাজ stash করো git stash git stash save "login form-এর অসম্পূর্ণ কাজ" # Untracked ফাইলও stash করো (new files) git stash -u git stash --include-untracked # সব stash দেখো git stash list # stash@{0}: On feature/login: login form-এর অসম্পূর্ণ কাজ # stash@{1}: On main: navbar update WIP # Stash-এর বিস্তারিত দেখো git stash show stash@{0} git stash show -p stash@{0} ← diff সহ
# Stash ফিরিয়ে আনো git stash pop ← সর্বশেষ stash (stash@{0}) আনো এবং মুছো git stash apply ← আনো কিন্তু মুছো না (একাধিক জায়গায় apply করতে) git stash apply stash@{1} ← নির্দিষ্ট stash # Stash মুছো git stash drop stash@{0} ← একটি মুছো git stash clear ← সব মুছো # Stash থেকে নতুন branch তৈরি করো git stash branch feature/login stash@{0} # নতুন branch-এ stash apply হবে

সম্পূর্ণ Stash Workflow

# ━━━━━━ কাজ করছো ━━━━━━ git checkout feature/login # ... login form-এ কাজ করছো ... # ━━━━━━ জরুরি ডাক! ━━━━━━ git stash save "login form WIP — validation বাকি" # ━━━━━━ hotfix করো ━━━━━━ git checkout main git pull git checkout -b hotfix/payment-crash # ... fix করো ... git commit -m "fix: payment null user crash সমাধান" git push origin hotfix/payment-crash # PR দাও বা direct merge করো # ━━━━━━ আগের কাজে ফিরে যাও ━━━━━━ git checkout feature/login git stash pop ← আগের অসম্পূর্ণ কাজ ফিরে আসবে!

🕰️ History — Linear, Non-Linear, HEAD ও Detached HEAD

🔷 HEAD কী?

HEAD হলো একটি pointer যা সবসময় নির্দেশ করে তুমি এখন কোন commit-এ আছো। সাধারণত HEAD একটি branch-এর শেষ commit-এ থাকে।

# HEAD-এর বর্তমান অবস্থা দেখো cat .git/HEAD # ref: refs/heads/main ← main branch-এর শেষে # HEAD মানে তোমার বর্তমান commit git log HEAD --oneline git show HEAD ← সর্বশেষ commit git show HEAD~1 ← তার আগের commit git show HEAD~2 ← আরো একটি পিছনে # Linear History (এক সরল লাইন): A → B → C → D (HEAD → main) # Non-Linear History (branch ও merge-এর পরে): A → B → C → F (HEAD → main) ↘ D → E ↗

Detached HEAD — বিপদজনক অবস্থা

# Detached HEAD হয় যখন তুমি সরাসরি commit checkout করো: git checkout a1b2c3d ← branch-এর নাম নয়, commit hash # তারপর git status দেখো: $ git status HEAD detached at a1b2c3d # মানে: তুমি কোনো branch-এ নেই! # এই অবস্থায় commit করলে branch ছাড়া হারিয়ে যাবে! # সমাধান ১: কোনো branch-এ ফিরে যাও git checkout main # সমাধান ২: এই commit থেকে নতুন branch তৈরি করো git checkout -b explore-old-version # সমাধান ৩: আধুনিক পদ্ধতি git switch main ← switch command clearly "branch শুধু"

🔄 Revert ও Reset — বিস্তারিত তুলনা

🔷 git reset-এর তিনটি Mode বিস্তারিত

Git Reset-এর তিনটি Mode visualize করে বোঝা: Commits: A → B → C (HEAD) তুমি এখন C-তে ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git reset --soft HEAD~1 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Repository: A → B (HEAD সরে গেল) Staging: C-র পরিবর্তন এখানে আছে ✅ Working: অপরিবর্তিত ✅ → ব্যবহার: commit message ঠিক করতে চাই, কোড রাখবো ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git reset HEAD~1 (--mixed, default) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Repository: A → B (HEAD সরে গেল) Staging: খালি ⚪ Working: C-র পরিবর্তন এখানে আছে ✅ → ব্যবহার: পরিবর্তন রেখে নতুনভাবে stage ও commit করবো ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ git reset --hard HEAD~1 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Repository: A → B (HEAD সরে গেল) Staging: খালি ⚪ Working: খালি ⚪ (C-র পরিবর্তন মুছে গেছে!) → ব্যবহার: সব বাতিল, পুরোপুরি B-তে ফিরে যাই

📐 Rebase — History পুনর্গঠন

Rebase হলো Git-এর সবচেয়ে শক্তিশালী ও complex feature। সঠিকভাবে ব্যবহার করলে history অনেক clean হয়।

🔷 Rebase vs Merge — পার্থক্য

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Merge-এর History (দুই branch-এর merge commit দৃশ্যমান): ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ main: A ─── B ─── C ─────────── G (merge commit) \ / feature: D ─── E ─ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Rebase-এর History (straight line, পরিষ্কার): ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ main: A ─── B ─── C ─── D' ─── E' feature (rebased): D ও E main-এর উপরে নতুনভাবে apply হয়েছে (D' ও E' = নতুন hash, কিন্তু একই পরিবর্তন)
# Feature branch-কে main-এর উপরে rebase করো git checkout feature/login git rebase main # Conflict হলে: git status ← কোন ফাইলে conflict? # ... conflict সমাধান করো ... git add fixed-file.html git rebase --continue # Rebase বাতিল করো git rebase --abort # main থেকে rebase (main আপ-টু-ডেট রাখতে) git fetch origin git rebase origin/main

🔷 Interactive Rebase — History পরিষ্কার করা

কাজ করার সময় অনেক ছোট commit হয়। PR দেওয়ার আগে সেগুলো পরিষ্কার করতে Interactive Rebase দারুণ।

# শেষ ৪টি commit interactive mode-এ দেখো git rebase -i HEAD~4 # Editor খুলবে: pick a1b2c3d feat: login form HTML pick b2c3d4e fix: typo in form label pick c3d4e5f style: login button রঙ pick d4e5f6g feat: form validation # Commands: # pick = এই commit রাখো (পরিবর্তন নেই) # reword = এই commit রাখো কিন্তু message পরিবর্তন করো # squash = আগের commit-এর সাথে মিলিয়ে দাও # fixup = squash-এর মতো, কিন্তু message নেবে না # drop = এই commit বাতিল করো # edit = এই commit-এ থামো এবং amend করো
# উদাহরণ: typo fix কে squash করো login-এর সাথে # আগে: pick a1b2c3d feat: login form HTML pick b2c3d4e fix: typo in form label ← এটা নিজে commit দরকার নেই pick c3d4e5f style: login button রঙ pick d4e5f6g feat: form validation # পরিবর্তন করো: pick a1b2c3d feat: login form HTML fixup b2c3d4e fix: typo in form label ← squash + message ফেলো pick c3d4e5f style: login button রঙ pick d4e5f6g feat: form validation # Save করলে b2c3d4e → a1b2c3d-এ মিলে যাবে
# git commit --amend # শেষ commit পরিবর্তন করতে: # Message ঠিক করো git commit --amend -m "feat: login form HTML — email ও password field" # ভুলে file add করতে ভুলেছিলে git add forgotten-file.css git commit --amend --no-edit # ⚠️ amend করলে commit hash পরিবর্তন হয় # Push হয়ে গেলে amend করো না
# git filter-branch / git filter-repo # সব history থেকে একটি ফাইল পুরোপুরি মুছতে # (sensitive data মুছার জন্য): # আধুনিক পদ্ধতি (git-filter-repo ইনস্টল করতে হবে) git filter-repo --path passwords.txt --invert-paths # ⚠️ এটি সব commit hash পরিবর্তন করে! # force push লাগবে এবং সবাইকে re-clone করতে হবে
⚠️ Rebase-এর সোনালী নিয়ম:
Public/shared branch-এ কখনো rebase করো না। main, develop, বা যেকোনো branch যা অন্যরা use করে — সেখানে rebase করলে অন্যদের history পরিবর্তন হয়ে যায় এবং বড় সমস্যা হয়।
শুধু তোমার নিজের local/private branch-এ rebase করো।

🏷️ Tagging ও GitHub Releases — Version চিহ্নিত করা

Software version release করার সময় Tag দিয়ে সেই commit চিরতরে চিহ্নিত করো।

🔷 Tag কী এবং কেন দরকার?

Tag হলো একটি নির্দিষ্ট commit-এর উপরে দেওয়া permanent label। Branch-এর মতো সরে যায় না। Software release করার সময় v1.0.0, v2.1.3 এভাবে tag দেওয়া হয়।

Tag ধরন বৈশিষ্ট্য কখন ব্যবহার
Lightweight Tag শুধু commit hash-এর alias, কোনো metadata নেই private, temporary marking
Annotated Tag Full object — tagger, date, message আছে public release-এর জন্য সঠিক
# Annotated tag তৈরি (release-এর জন্য সঠিক পদ্ধতি) git tag -a v1.0.0 -m "Version 1.0.0 — প্রথম stable release নতুন features: - User authentication - Product listing - Shopping cart - Payment integration Bug fixes: - Mobile navbar rendering - Email validation" # Lightweight tag (দ্রুত, private) git tag v1.0.0-beta # সব tag দেখো git tag git tag -l "v1.*" ← v1 দিয়ে শুরু সব tag # একটি tag-এর বিবরণ দেখো git show v1.0.0 # পুরনো commit-এ tag দাও git tag -a v0.9.0 a1b2c3d -m "Beta release"
# Tag GitHub-এ push করো git push origin v1.0.0 ← একটি tag git push origin --tags ← সব tag # Tag মুছো git tag -d v1.0.0-beta ← local git push origin --delete v1.0.0-beta ← remote # Tag-এ checkout করো (Detached HEAD হবে) git checkout v1.0.0 # → "You are in 'detached HEAD' state"

Semantic Versioning (SemVer) — নিয়ম মেনে Version

Format: MAJOR.MINOR.PATCH উদাহরণ: v2.4.1 MAJOR (2): Breaking changes — পুরনো code কাজ নাও করতে পারে MINOR (4): নতুন feature — backward compatible PATCH (1): Bug fix — backward compatible উদাহরণ: v1.0.0 → প্রথম stable release v1.1.0 → নতুন feature যোগ (backward compatible) v1.1.1 → bug fix v2.0.0 → API পরিবর্তন হয়েছে (breaking change) Pre-release: v2.0.0-alpha.1 → early testing v2.0.0-beta.3 → broader testing v2.0.0-rc.1 → release candidate

GitHub Releases

# GitHub Releases = Tag + Release Notes + Binary attachments # GitHub-এ Release তৈরি: # Releases → Draft a new release # → Tag: v1.0.0 (নতুন বা বিদ্যমান tag) # → Release title: "Version 1.0.0 — First Stable Release" # → Description: changelog, new features, breaking changes # → Attachments: compiled binary, zip file ইত্যাদি # → Publish release ক্লিক করো

🪝 Git Hooks — Automated Scripts

Git Hooks দিয়ে commit, push ইত্যাদি ঘটনার সময় স্বয়ংক্রিয়ভাবে script চালানো যায়।

🔷 Git Hooks কী এবং কেন?

Git Hooks হলো automatic scripts যা নির্দিষ্ট Git ঘটনার আগে বা পরে চলে। এই দিয়ে quality control enforce করা যায় — যেমন commit-এর আগে lint check, push-এর আগে tests চালানো।

Hook নাম কখন চলে সাধারণ ব্যবহার
pre-commit commit শুরু হওয়ার আগে Lint check, code formatting, secret detection
commit-msg Commit message লেখার পরে Commit message format যাচাই (conventional commits)
pre-push Push-এর আগে Tests run করা, build যাচাই
post-checkout checkout-এর পরে Dependencies install notify করা
post-update Server-এ push-এর পরে (server-side) Notification পাঠানো, CI trigger করা
# Hooks কোথায় থাকে? ls .git/hooks/ # pre-commit.sample # commit-msg.sample # pre-push.sample # ... (সব sample — .sample suffix মুছলে active হয়) # একটি pre-commit hook তৈরি করো: cat > .git/hooks/pre-commit << 'HOOK' #!/bin/sh # Commit-এর আগে এই script চলবে echo "🔍 Code quality check চলছে..." # ESLint check if command -v npm >/dev/null; then npm run lint 2>&1 if [ $? -ne 0 ]; then echo "❌ Lint errors আছে! Commit বাতিল।" echo "🔧 'npm run lint:fix' চালাও এবং আবার try করো।" exit 1 fi fi echo "✅ সব check pass! Commit করা হচ্ছে..." exit 0 HOOK # Executable করো (অবশ্যই) chmod +x .git/hooks/pre-commit

Husky — Node.js Projects-এ সহজ Hook Management

# Husky ইনস্টল করো npm install --save-dev husky # Husky initialize করো npx husky init # .husky/pre-commit ফাইল তৈরি হবে: echo "npm run lint" > .husky/pre-commit echo "npm test" >> .husky/pre-commit # .husky/commit-msg hook (conventional commits check): echo 'npx --no -- commitlint --edit "$1"' > .husky/commit-msg # এখন: # - প্রতিটি commit-এর আগে lint চলবে # - Lint fail করলে commit হবে না

⚡ GitHub Actions — CI/CD Automation

GitHub Actions দিয়ে code push হলে স্বয়ংক্রিয়ভাবে test, build ও deploy করা যায়।

🔷 GitHub Actions কী?

GitHub Actions হলো GitHub-এর built-in CI/CD (Continuous Integration / Continuous Deployment) platform। তুমি YAML ফাইলে define করো কখন কী করবে — push হলে, PR দিলে, বা schedule-এ।

উপাদান বিবরণ
Workflow একটি YAML ফাইল যেখানে সব কিছু define করা আছে
Trigger (on:) কখন workflow চলবে — push, PR, schedule, manual
Job Workflow-এর একটি অংশ যা Runner-এ চলে
Step Job-এর প্রতিটি ধাপ
Action Reusable script বা GitHub Marketplace থেকে
Runner যে machine-এ workflow চলে (ubuntu, windows, mac)

🔷 Git Submodules — Repository-র ভিতরে Repository

বড় project-এ কখনো অন্য Git repository কে "sub-project" হিসেবে যোগ করতে হয়। উদাহরণ: তোমার main project-এ একটি shared UI library যোগ করা।

# Submodule যোগ করো git submodule add git@github.com:org/ui-library.git libs/ui # Submodule সহ clone করো git clone --recurse-submodules git@github.com:username/project.git # বিদ্যমান clone-এ submodule initialize করো git submodule init git submodule update # সব submodule আপডেট করো git submodule update --remote

🔷 Git Bisect — Bug কোন Commit-এ এলো?

হাজার commit-এর মধ্যে কোনটিতে bug এলো তা binary search-এর মাধ্যমে খুঁজে বের করা।

# Git bisect শুরু করো git bisect start # বর্তমান commit buggy বলো git bisect bad HEAD # কোনো পুরনো ভালো commit বলো git bisect good v1.0.0 # Git মাঝের commit checkout করবে # → তুমি test করো, bug আছে কিনা git bisect bad ← bug আছে git bisect good ← bug নেই # Git আবার মাঝের দিকে যাবে — binary search! # ৩০ commits = মাত্র ৫ বার check করলেই পাওয়া যাবে # Bug পেয়ে গেলে git bisect reset ← HEAD-এ ফিরে যাও

🔷 Git LFS — বড় ফাইল Handle করা

Git সাধারণত text file-এর জন্য তৈরি। বড় binary file (PSD, video, database dump) commit করলে repository অনেক বড় হয়ে যায়। Git LFS (Large File Storage) এই সমস্যা সমাধান করে।

# Git LFS install করো git lfs install # কোন ধরনের ফাইল LFS-এ রাখবে git lfs track "*.psd" git lfs track "*.mp4" git lfs track "*.zip" git lfs track "*.pdf" # .gitattributes তৈরি হবে — এটা commit করো git add .gitattributes git commit -m "chore: Git LFS tracking configure করা হয়েছে" # LFS-tracked ফাইল দেখো git lfs ls-files # LFS status git lfs status

🛠️ Git Patch, Worktree ও Attributes

Git-এর কিছু কম পরিচিত কিন্তু কাজের advanced feature।

🔷 Git Patch — Email বা File দিয়ে Changes শেয়ার করো

Git Patch দিয়ে changes-কে একটি ফাইলে রপ্তানি করা যায় এবং অন্য repository-তে apply করা যায়। GitHub access ছাড়া changes শেয়ার করার পুরনো কিন্তু কার্যকর পদ্ধতি।

# শেষ ১টি commit-এর patch তৈরি করো git format-patch -1 HEAD # শেষ ৩টি commit-এর patch git format-patch -3 HEAD # নির্দিষ্ট commit range git format-patch main..feature-branch # Patch apply করো git apply 0001-feat-add-search.patch # am দিয়ে apply (commit সহ) git am 0001-feat-add-search.patch # Patch preview দেখো (apply না করে) git apply --check 0001-feat-add-search.patch

🔷 Git Worktree — একসাথে একাধিক Branch-এ কাজ করো

সাধারণত একটি repository-তে একবারে একটাই branch checkout করা যায়। Git Worktree দিয়ে একই repository-র একাধিক branch আলাদা folder-এ একসাথে open রাখা যায়।

# নতুন worktree তৈরি করো (branch সহ) git worktree add ../project-hotfix hotfix/login-bug # বিদ্যমান branch-এর worktree git worktree add ../project-feature feature/search # সব worktree দেখো git worktree list # Worktree সরাও git worktree remove ../project-hotfix # Worktree prune করো (deleted folder cleanup) git worktree prune
✅ Worktree কখন কাজে লাগে:

Feature develop করতে করতে hotfix লাগলে → stash না করে আলাদা folder-এ fix করো। দুটো version একসাথে compare করতে হলে → দুটো folder পাশাপাশি খোলো।

🔷 Git Attributes — ফাইল-ভিত্তিক নিয়ম

.gitattributes ফাইলে নির্দিষ্ট ফাইল বা ফাইল-type-এর জন্য বিশেষ নিয়ম সেট করা যায়।

# .gitattributes ফাইল তৈরি করো # Line ending normalize করো (Windows/Mac/Linux) * text=auto *.bat text eol=crlf *.sh text eol=lf # Binary ফাইল mark করো (diff দেখাবে না) *.png binary *.jpg binary *.pdf binary *.zip binary # Merge strategy নির্দিষ্ট করো package-lock.json merge=ours # Diff driver কাস্টম করো *.md diff=markdown # Export-ignore (zip/archive-এ বাদ দাও) .github export-ignore tests/ export-ignore *.md export-ignore
✅ .gitattributes-এর সবচেয়ে গুরুত্বপূর্ণ ব্যবহার:

Team-এ Windows ও Mac ডেভেলপার একসাথে কাজ করলে line ending conflict হয়। * text=auto দিলে Git automatically normalize করে।
